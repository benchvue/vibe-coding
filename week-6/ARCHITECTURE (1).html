Engine Architecture

A single-file, dependency-light browser racing game built on a deterministic per-frame game loop. Input flows one direction into an authoritative physics model, which drives every downstream system.

02  Layers in detail

INPUT

Input layer

Three independent sources, normalised to the same intent values. The keyboard handler tracks raw arrow-key booleans. The gamepad reader polls the Gladiator NXT EVO each frame, applies a 0.12 deadzone then an expo curve so a resting stick never creeps the car. The GLTFLoader fetches the GLB once, scales and centres it, and rigs the four wheels by mesh-name (spinning them, steering the fronts) while leaving brake calipers fixed.

SIM CORE

Vehicle physics model

The single source of truth. An arcade bicycle model integrates throttle, braking, steering and a low-speed turn boost into speed, heading and world position. Every other system reads this state; none read each other. This is why features were added incrementally without regressions.

SYSTEMS

Per-frame systems

Four consumers of the physics state. Collision hit-tests obstacles and runs the crash/rollover animation. The Three.js scene rebuilds the track ribbon, positions the car and chase camera. The HUD draws the speedometer, mini-map and score. Web Audio scales engine pitch and filter cutoff with speed and throttle.

LOOP

Game loop

The orchestrator. Each requestAnimationFrame tick computes a clamped dt and runs step(dt) in a fixed order, then renders the frame. Deterministic ordering is what keeps physics, collision and rendering in sync.

STATE

Persistence

A thin edge. The high score is written to localStorage when beaten and restored on load, wrapped in try/catch so sandboxed contexts degrade gracefully.

03  Frame execution order

Every step(dt) runs the same sequence. Order matters: input must be sampled before physics, collisions resolved before the scene is drawn, and the HUD updated last so it reflects the final frame state.

  1. Sample input — merge keyboard and gamepad into throttle / brake / steer intents.
  2. Step physics — integrate the bicycle model; update speed, heading, position, wheel spin.
  3. Resolve collisions — hit-test obstacles; if hit, enter the crash + rollover state.
  4. Update scene — move the car group, advance the chase camera, spin and steer wheels.
  5. Update audio — retune engine oscillators and tyre noise from speed and throttle.
  6. Update HUD — redraw speedometer needle, mini-map marker, score and high score.
  7. Render — one Three.js draw call, then schedule the next frame.

04  Design principles

One-directional data flow. Input → physics → systems → render. No system reads another system's internals; they all read the physics model. This containment is the reason the GLB loader, mini-map, crash rollover and low-speed steering could each be added without touching the others.

Single source of truth. The vehicle physics object holds the only authoritative state. Rendering, audio and the HUD are pure projections of it — never the other way around.

Single file, no build step. Three.js and the GLTF loader come from a CDN; everything else is vanilla JS in one HTML file. It runs by opening the file (served over HTTP for the Gamepad API and GLB fetch).
